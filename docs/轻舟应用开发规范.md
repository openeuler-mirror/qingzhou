# 轻舟应用开发规范

## 概述

轻舟对业务系统的管控是通过轻舟代理调度对应的“轻舟应用”来完成的。
轻舟应用是基于“轻舟 API”开发的 Java 应用程序。
轻舟 API 为各类业务系统提供统一的开发规范，包括接口、UI、集成方式等，最终实现多品类业务系统的标准化、一致化管理。

## 快速开始

📌 参考示例：`apps/qingzhou-app-demo`

1. 新建项目，在 `qingzhou/apps` 目录下新建 Maven 子模块。
2. 编写应用入口类，入口类会在轻舟启动时被调用。
    - 创建类，实现 `QingzhouApp` 接口。
    - 添加 `@App`、`@Menu` 注解，配置元数据。
    - 重写 `start()` 方法，定制启动逻辑。
3. 编写业务模块类，模块类对应业务实体，在管控台上体现为功能菜单。
    - 创建类，继承 `ModelBase`。
    - 添加 `@Model` 注解，配置元数据。
    - 根据业务需求，实现相应的能力接口，如`List`、`Show`、`Add`、`Update`、`Delete`、`Monitor`等。
4. 添加业务模块字段，模块字段对应业务数据，在管控台上体现为表单字段。
    - 创建属性，如 `public String id`。
    - 添加 `@ModelField` 注解，配置元数据。
5. 添加业务模块操作，模块操作对应 REST 接口，在管控台上体现为按钮。
    - 创建方法，如 `public void myMethod(Request request){}`，注意方法的参数有且只能有一个 Request 类型。
    - 添加 `@ModelAction` 注解，配置元数据。
6. 部署
    - 在新建项目下执行 `mvn clean install` 完成编译。
    - 将生成的 jar 包放入`${轻舟产品包}/instances/default/apps` 目录下。
    - 重启轻舟，访问管控台，可看到开发的应用。

## 自动发现

轻舟应用是用于管理目标软件的，在实施过程中，轻舟应用和目标软件通常是独立部署的，轻舟应用需要等待目标软件部署完成后才能正常工作，这就需要轻舟应用具备自动发现目标软件后自动完成自身部署的能力。
在轻舟平台上，轻舟应用可以通过在 `META-INF/MANIFEST.MF` 中配置 `Qingzhou-Detection-*` 的方式来启用此功能。

### 配置方式

在开发应用的 Maven 项目的 `pom.xml` 中配置 `maven-jar-plugin`，通过 `manifestEntries` 注入自动发现相关配置，以便在项目打包时写入
`META-INF/MANIFEST.MF`，示例如下：

```xml

<plugin>
    <artifactId>maven-jar-plugin</artifactId>
    <configuration>
        <archive>
            <manifestEntries>
                <Qingzhou-Detection-Enabled>true</Qingzhou-Detection-Enabled>
                <Qingzhou-Detection-Feature-Files>bin/java,bin/java.exe</Qingzhou-Detection-Feature-Files>
                <Qingzhou-Detection-Env-Vars>Demo_HOME</Qingzhou-Detection-Env-Vars>
                <Qingzhou-Detection-Scan-Roots>user.home,/usr/local,/opt</Qingzhou-Detection-Scan-Roots>
                <Qingzhou-Detection-Process-Names>bin/java</Qingzhou-Detection-Process-Names>
            </manifestEntries>
        </archive>
    </configuration>
</plugin>
```

### 参数详解

| 参数名                                | 必填 | 默认值     | 说明                                                                                                          |
|------------------------------------|----|---------|-------------------------------------------------------------------------------------------------------------|
| `Qingzhou-Detection-Enabled`       | 否  | `false` | 是否启用应用的自动发现功能。未启用时，后续所有配置参数均无效。                                                                             |
| `Qingzhou-Detection-Feature-Files` | 是  | -       | 检测特征文件列表，逗号分隔。该特征文件是从待检测目标软件安装目录开始的**相对路径**。任意一个特征文件匹配成功则结束探测。应用程序可通过 `AppContext.getDetectedPath()` 获取发现的路径。 |
| `Qingzhou-Detection-Env-Vars`      | 否  | -       | 环境变量列表，逗号分隔。一些目标软件通过环境变量设置了安装目录（如 `JAVA_HOME`、`M2_HOME`），配置此项可提高探测效率。                                         |
| `Qingzhou-Detection-Scan-Roots`    | 否  | -       | 目录扫描根路径列表，逗号分隔。支持特殊变量 `user.home`（当前用户主目录）。仅扫描配置目录及其一级子目录。                                                  |
| `Qingzhou-Detection-Process-Names` | 否  | -       | 进程名称列表，逗号分隔。如果未能通过环境变量和目录扫描找到目标软件安装目录，则会从当前正在运行的所有程序列表中继续探测。                                                  |

### 获取探测结果

应用启动后，可通过以下方式获取探测到的路径：

```java
public class MyApp implements QingzhouApp {
    @Override
    public void start(AppContext context) {
        String detectedPath = context.getDetectedPath();
        if (detectedPath != null) {
            // 目标软件已发现，执行初始化逻辑
            System.out.println("已发现目标系统: " + detectedPath);
        } else {
            // 目标软件未发现，应用处于等待状态
        }
    }
}
```

### 最佳实践

1. **特征文件选择**：选择目标软件安装目录下**唯一且稳定**的文件作为特征标识，如 `bin/startup.sh`、`lib/core.jar` 等。
2. **环境变量优先**：对于通过环境变量安装的软件（如 JDK、Maven），优先配置 `Qingzhou-Detection-Env-Vars`，可显著提高探测效率。
3. **扫描路径精简**：避免配置过深或过广的扫描路径，建议限定在 `/opt`、`/usr/local` 等常见安装目录。
4. **进程名称精确**：进程名称应包含可识别的关键字，如 `java`、`nginx`、`mysql` 等，避免过于宽泛。
5. **不同平台适配**：通过配置多个特征文件（如 `bin/java`、`bin/java.exe`）支持跨平台部署。
